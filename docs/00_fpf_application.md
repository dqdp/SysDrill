# Applying First Principles Framework to System Design Trainer

## Зачем здесь вообще FPF

Для этого проекта FPF полезен не как “словарь сложных терминов”, а как способ
не перепутать:

- **что является системой**, а что только интерфейсом;
- **где живёт смысл**, а где живёт presentation;
- **что является знанием**, а что является runtime-поведением;
- **что является оценкой**, а что является персонализацией;
- **что является фактом взаимодействия**, а что является интерпретацией.

## Проект от первых принципов

Мы строим не “сайт с карточками по system design”.

Мы строим **систему тренировки навыка системного дизайна**, которая должна:

1. помогать вспоминать и связывать базовые концепты;
2. учить видеть требования и ограничения;
3. тренировать decomposition и trade-off reasoning;
4. давать подсказки, не разрушая effortful recall;
5. симулировать mock interview;
6. собирать interaction history и улучшать траекторию обучения.

Из этого следует, что основным объектом проектирования является
**learning system**, а не UI.

## Что FPF требует сделать явным

### 1. Bounded contexts
Нельзя смешивать в один комок:

- контент;
- дизайн обучения;
- runtime сессии;
- оценку;
- рекомендательную систему;
- взаимодействие через UI/voice.

### 2. Structured artefacts
Вместо “пусть LLM как-нибудь поймёт” нужны явные артефакты:

- Concept
- Pattern
- Scenario
- Hint Ladder
- Rubric
- Session
- Interaction Event
- Learner Profile
- Recommendation Decision
- Review Report

### 3. Hand-offs
Каждый переход между контекстами должен быть описан как hand-off contract:
какие данные переходят, кто за них отвечает и что считается валидным.

### 4. Many lawful views
Одно и то же решение должно быть выражаемо как минимум в трёх поверхностях:

- **product view**
- **engineering view**
- **assurance view**

Это предотвращает ситуацию, когда product deck, engineering docs и scoring logic
описывают по сути три разные системы.

## Главная корректировка после последнего замечания

В раннем варианте было недостаточно явно выделено, что:

- хранение истории взаимодействий;
- построение learner model;
- формирование следующего шага обучения

— это отдельная система, а не просто persistence.

Поэтому в этом пакете recommendation выделен в отдельный контекст
**Learning Intelligence**.

## FPF-совместимые принципы для проекта

1. **Content is not prompt text**  
   Канонический контент должен жить вне промптов.

2. **Runtime is not recommendation**  
   Оркестрация сессии и долгосрочная траектория обучения — разные вещи.

3. **Evaluation is not memory**  
   Оценка одной сессии не равна learner profile.

4. **Claims need evidence**  
   Любая рекомендация должна иметь trace:
   почему именно это следующее действие было выбрано.

5. **Different roles need different views**  
   Основатель, инженер и reviewer качества не должны читать один и тот же документ.

## Минимальный набор публикационных поверхностей

- Spec / architecture docs для инженерной реализации.
- Product docs для roadmap и scope control.
- Assurance docs для проверки качества и explainability.
- ADR для долговечных решений.
- YAML schemas для реализации storage / DTO / contracts.

## Что НЕ надо делать

- строить всё вокруг одной mega-prompt;
- прятать recommendation logic внутри LLM;
- путать knowledge source с UI-карточкой;
- вводить voice-first до доказательства core learning loop;
- делать opaque scoring без rubric и justification;
- хранить только финальные score без interaction trail.

## Вывод

FPF здесь нужен как дисциплина:
**развести ответственность, зафиксировать смысл, сделать hand-offs и построить
несколько согласованных видов одной системы**.
